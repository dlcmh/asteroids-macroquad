Chapter 5a

Avoiding Asymmetry Typos

One wrong number can make your rocket's flame shoot out sideways. Here's how experts never make that mistake.

The Bug That's Too Easy to Write

When you draw a symmetrical shape like a ship or a flame, you're usually computing points on both sides of a centre line. In code, that means two very similar lines:

let left = Vec2::new(
    base.x + (back + 1.5).cos() * spread,
    base.y + (back + 1.5).sin() * spread,
);
let right = Vec2::new(
    base.x + (back + 1.5).cos() * spread,
    base.y + (back - 1.5).sin() * spread,  // ← see it?
);

back + 1.5 on x, but back - 1.5 on y. The left and right edges of the flame don't rotate together. At some angles the triangle collapses into a skinny line, spraying flame out the side of your ship instead of behind it.

This is the asymmetry typo. It happens because you're typing the same pattern four times (something.cos(), something.sin() × 2 points) and your brain autocompletes the third line from the first without checking.

What the bug looks like

Pattern 1: One Helper to Rule Them All

Write a tiny helper that converts an angle and distance into a point. Then every shape uses it, and there's only one place where cos and sin appear together:

/// Return a point `dist` pixels from `centre` in direction `angle`.
fn offset(centre: Vec2, angle: f32, dist: f32) -> Vec2 {
    Vec2::new(
        centre.x + angle.cos() * dist,
        centre.y + angle.sin() * dist,
    )
}

Now drawing becomes a list of almost-English statements. Hard to get wrong:

fn draw_flame(centre: Vec2, nose_angle: f32) {
    let back = nose_angle + PI;          // opposite direction

    let base  = offset(centre, back, 8.0);
    let tip   = offset(base,   back, rand::gen_range(8.0, 20.0));
    let left  = offset(base,   back + 1.5, rand::gen_range(3.0, 7.0));
    let right = offset(base,   back - 1.5, rand::gen_range(3.0, 7.0));

    draw_triangle(tip, left, right, ORANGE);
}

Key insight: offset packs the cos/sin pair into one function. You type it once, test it once, and never think about it again. Every point derived from it is guaranteed internally consistent.

Pattern 2: Design the Shape in Local Space

Instead of computing each vertex from scratch, describe the shape relative to the ship's centre and rotate the whole thing at once:

// Vertices of the ship in "local space":
//   (forward distance, sideways distance)
const HULL: [(f32, f32); 3] = [
    ( 20.0,  0.0),   // nose: straight ahead
    (-12.0, -8.0),   // left wing: back & left
    (-12.0,  8.0),   // right wing: back & right
];

Then apply a rotation matrix — one formula that spins every point by the ship's angle:

fn draw_hull(centre: Vec2, angle: f32) {
    let cos = angle.cos();
    let sin = angle.sin();

    // Rotate every local vertex by the ship's angle
    let points: [Vec2; 3] = HULL.map(|(dx, dy)| {
        Vec2::new(
            centre.x + dx * cos - dy * sin,
            centre.y + dx * sin + dy * cos,
        )
    });

    draw_triangle(points[0], points[1], points[2], WHITE);
}

Same local shape, any rotation

The rotation matrix handles every angle. No per-vertex trigonometry, no asymmetry bugs.

Key insight: This is the number-one technique professional game devs use. You describe the shape once in local coordinates (centred at 0,0), and the rotation+translation code handles the rest. Adding a new vertex doesn't require any new trigonometry.

Pattern 3: Give Every Angle a Name

When you're computing two points from two similar angles, pull each angle into a named variable before using it:

// Instead of:
let left = Vec2::new(
    base.x + (back + 1.5).cos() * spread,
    base.y + (back + 1.5).sin() * spread,
);
let right = Vec2::new(
    base.x + (back + 1.5).cos() * spread,   // copy-paste: left over
    base.y + (back - 1.5).sin() * spread,   // but changed here — bug!
);

// Name them:
let left_angle  = back + 1.5;
let right_angle = back - 1.5;

let left  = offset(base, left_angle,  spread);
let right = offset(base, right_angle, spread);

// Now the asymmetry is visible in the variable names.

Key insight: A named variable changes the visual pattern from "four nearly identical lines" to "two clearly different angles, each used consistently." Code reviews catch mismatched variable names instantly. They don't catch mismatched inline expressions.

Pattern 4: Mirror One Side

If your shape is perfectly symmetrical about the centre line, compute one wing and derive the other by reflecting it:

// Compute the left wing...
let left = offset(centre, angle - SWEEP, HALF_BASE);

// ...and mirror it across the centre to get the right wing.
let right = Vec2::new(
    2.0 * centre.x - left.x,
    2.0 * centre.y - left.y,
);

This works because for any point on the left side, the mirrored right side is the same distance from centre but on the opposite side. No second angle computation at all.

Key insight: Reflection is foolproof — there's literally no second angle to get wrong. The trade-off is it only works for shapes that are perfectly symmetrical. Most game objects are.

How to Catch These Bugs Fast

🔄 The Spin Test

Before adding gameplay, add a simple rotation loop that spins the ship 360° automatically. Watch the flame and the wings. If the flame wobbles or the ship seems to change shape as it turns, you have an asymmetry bug. This catches the problem in two seconds.

⭕ The Centre Dot

Draw a small dot at the ship's centre and a circle at the flame base. Now you can see visually whether the flame base is actually centred behind the ship. Sometimes the offset is correct but the base position is wrong.

📐 Print the Angles

Add a quick println!("left angle: {}, right angle: {}", left_angle, right_angle); and check they're the same magnitude with opposite signs. This catches the typo in the terminal before you even look at the screen.

Which Pattern Should You Use?

🆕 Beginners: Use Pattern 1 (offset helper)

Write the offset function once. Every shape then becomes a list of ordinary-looking lines. It's the smallest change from the code you're already writing and eliminates the most common bug instantly.

📐 Intermediate: Use Pattern 2 (local offsets + rotation matrix)

Once you're comfortable with the math, this is the professional standard. It scales to complex shapes (8-gon asteroids, detailed ships) with zero additional trigonometry per vertex.

🧠 Always: Use Patterns 3 & 4 (named angles, mirror)

These are habits that cost nothing and make bugs visible. A named variable is easier to read and harder to mistype. Mirroring eliminates the second computation entirely.

🎯 Before you write another triangle, write offset.

It's three lines that will save you from debugging a flame that shoots sideways for twenty minutes. Trust me — I'm writing this chapter because I made exactly that mistake.
