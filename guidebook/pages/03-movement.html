Chapter 3

Movement & Game Loops

How things fly through space. No asteroids yet—just pure motion.

The Two Vectors That Run Everything

Every moving object in our game needs just two pieces of information:

📍 Position

Where is it right now?

Vec2::new(x, y)

⚡ Velocity

How fast and in what direction is it moving?

Vec2::new(dx, dy) 
struct Ship {
    position: Vec2,  // Where on screen
    velocity: Vec2,  // Pixels per second
}

// Starting at center, not moving
let ship = Ship {
    position: Vec2::new(screen_width() / 2.0, screen_height() / 2.0),
    velocity: Vec2::new(0.0, 0.0),
};

Velocity of (100.0, 0.0) means "move 100 pixels right every second." Velocity of (0.0, -50.0) means "move 50 pixels up every second" (remember: Y goes down, so negative Y is up!).

Euler Integration (The Magic Update Step)

Every frame, we update position using velocity. This is called Euler integration—named after the mathematician Leonhard Euler. Don't let the fancy name scare you:

new_position = old_position + velocity × delta_time

That's it. That's the whole thing.

Why multiply by delta_time?

Computers run at different speeds. If we just did position += velocity every frame:

  • On a 30 FPS machine: moves 30 × velocity per second
  • On a 60 FPS machine: moves 60 × velocity per second
  • On a 144 FPS machine: moves 144 × velocity per second (way too fast!)

Delta time (dt) is the actual time elapsed since the last frame. By multiplying, we make movement frame-rate independent.

// Inside the game loop
let dt = get_frame_time();  // macroquad gives us this

// Move the ship
ship.position.x += ship.velocity.x * dt;
ship.position.y += ship.velocity.y * dt;

// Or using Vec2 magic:
ship.position += ship.velocity * dt;

🧠 Mental Model

Imagine you're driving a car. Your position is where the car is. Your velocity is how fast the speedometer reads. Delta time is how long you've been driving at that speed. Distance traveled = speed × time. That's all Euler integration is.

Screen Wrapping (Asteroids' Signature)

In Asteroids, nothing ever stops. When you fly off one edge, you appear on the opposite side. This creates the feeling of infinite space.

fn wrap_position(pos: &mut Vec2, screen_w: f32, screen_h: f32) {
    if pos.x < 0.0 {
        pos.x += screen_w;  // Wrap right
    }
    if pos.x > screen_w {
        pos.x -= screen_w;  // Wrap left
    }
    if pos.y < 0.0 {
        pos.y += screen_h;  // Wrap down
    }
    if pos.y > screen_h {
        pos.y -= screen_h;  // Wrap up
    }
}

⚠️ Common Pitfall

Don't use % (modulo) in Rust for negative numbers—it behaves differently than you might expect. The if approach above is clearer and always correct.

Putting It All Together

Here's a complete program. Copy this into your main.rs and run it:

use macroquad::prelude::*;

struct Ship {
    position: Vec2,
    velocity: Vec2,
}

fn wrap_position(pos: &mut Vec2, screen_w: f32, screen_h: f32) {
    if pos.x < 0.0 { pos.x += screen_w; }
    if pos.x > screen_w { pos.x -= screen_w; }
    if pos.y < 0.0 { pos.y += screen_h; }
    if pos.y > screen_h { pos.y -= screen_h; }
}

#[macroquad::main("Movement Demo")]
async fn main() {
    let mut ship = Ship {
        position: Vec2::new(screen_width() / 2.0, screen_height() / 2.0),
        velocity: Vec2::new(100.0, 50.0), // Moving right and down
    };

    loop {
        let dt = get_frame_time();
        clear_background(BLACK);

        // Update: Euler integration
        ship.position += ship.velocity * dt;

        // Update: Screen wrapping
        wrap_position(&mut ship.position, screen_width(), screen_height());

        // Draw the ship (a circle for now)
        draw_circle(ship.position.x, ship.position.y, 15.0, YELLOW);

        // Draw velocity vector as a line
        draw_line(
            ship.position.x, 
            ship.position.y,
            ship.position.x + ship.velocity.x * 0.1,
            ship.position.y + ship.velocity.y * 0.1,
            2.0,
            RED
        );

        next_frame().await;
    }
}

Run it. You should see a yellow circle drifting across the screen and wrapping around the edges. The red line shows its velocity direction.

Try changing the initial velocity values. What happens with (0.0, 0.0)? What about (-100.0, 0.0)?

Adding Keyboard Control

A drifting ship is cool, but we want to fly it. Let's add arrow key controls:

// Add this before the Euler integration step
let thrust_power = 200.0; // Pixels per second²

if is_key_down(KeyCode::Right) {
    ship.velocity.x += thrust_power * dt;
}
if is_key_down(KeyCode::Left) {
    ship.velocity.x -= thrust_power * dt;
}
if is_key_down(KeyCode::Up) {
    ship.velocity.y -= thrust_power * dt; // Negative is UP
}
if is_key_down(KeyCode::Down) {
    ship.velocity.y += thrust_power * dt; // Positive is DOWN
}

💡 Why "thrust_power * dt"?

Velocity changes over time, just like position. We're doing Euler integration on velocity too! Thrust adds to velocity, velocity adds to position. This is acceleration.

Space Friction (Optional but Nice)

In real space, there's no friction—objects drift forever. But for a game, a tiny bit of drag makes the ship feel more controllable:

// Apply slight drag each frame
let drag = 0.99; // Multiply velocity by this each frame
ship.velocity *= drag;

At 60 FPS, a drag of 0.99 means velocity halves in about 1.1 seconds. Try different values: 0.995 for less drag, 0.98 for more.

Key Takeaways

Euler Integration: position += velocity * dt

The foundation of all game physics. Simple, effective, good enough for us.

Delta Time: Use get_frame_time()

Makes your game run the same speed on any computer.

Screen Wrapping: Check all four edges

Creates the feeling of infinite space with zero extra complexity.

Vec2 Operations: Add, multiply, they just work

Vector math makes game code clean and readable.
